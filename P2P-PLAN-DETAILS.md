## Сетевая модель (каналы WebRTC)

Топология: клиент ↔ Мастер — сигнальный WebSocket (только координация);
клиент ↔ хост — `RTCPeerConnection` с **двумя** DataChannel.

**Почему не «снапшоты по unreliable», как в ранней версии плана:** кадры v3
несут одноразовые события (`w1`/`w2`/`w2e`, создания/удаления сущностей,
clear/shake камеры), рассчитанные на гарантированную доставку WS/TCP. Потеря
кадра на unreliable-канале = навсегда потерянный взрыв или несозданный танк.

- **`meta`** (reliable-ordered): весь существующий JSON-протокол
  `[portId, payload]` (порты `src/config/wsports.js`: 0–4, 6–17) без изменений
  + бинарные кадры, **содержащие события**. Head-of-line blocking приемлем:
  событийные кадры редки.
- **`state`** (unreliable-unordered: `ordered: false, maxRetransmits: 0`):
  чисто позиционные бинарные кадры (`m1`/`c1`/`c2` + камера + player-блок +
  `serverTime`/`seq`) — их потеря компенсируется следующим кадром.
- Классификация на хосте при упаковке: кадр содержит событийные блоки → `meta`,
  иначе → `state`.
- RTT ping/pong — по `state` (честный замер без ретрансмиссий; потери уже
  учитывает логика `maxMissedPings`).

Клиентская сторона: буфер `SnapshotInterpolator` переводится с «push в конец»
(корректно только при TCP-порядке) на **вставку по `seq`** с дедупликацией и
отбрасыванием дублей; события из опоздавшего reliable-кадра, чей `serverTime`
уже позади `renderTime`, выдаются немедленно при вставке — «ровно один раз»
сохраняется. Формат кадра v3 не меняется.

### Этап 0: Снятие киллер-рисков (go/no-go перед жёсткой заменой)

**Цель:** Дешёвыми PoC проверить допущения, на которых держится архитектура, ДО
демонтажа рабочего сервера. Пока главный спайк не пройден — код не трогаем.

**Организация:** код спайков — в `_spikes/` (конвенция проекта: `_`-директории
не коммитятся); результаты (go/no-go, замеры, выводы) — в разделе «Результаты
Этапа 0» файла [P2P-PLAN.md](P2P-PLAN.md).

**0.1. Spike A — Пропускная способность WebRTC + upload хоста (главный go/no-go)**

- Замерить фактический размер бинарного кадра (`SnapshotPacker.packBody`,
  `src/lib/snapshotCodec.js`) для типичной сцены. Априорная оценка: заголовок +
  камера + player-блок ≈ 60 Б, танк ≈ 40 Б → кадр на 8 танков ≈ 400–500 Б;
  30 Гц × 7 исходящих потоков + оверхед DTLS/SCTP ≈ 1 Мбит/с — домашний uplink
  (~10 Мбит/с) проходит с большим запасом, узкое место скорее CPU/джиттер.
- Прототип: один браузер-хост шлёт синтетические кадры этого размера через
  `RTCDataChannel` (два канала по сетевой модели) на N подключений на 30 Гц.
- Замерить деградацию при 4/8/12 игроках на типичном домашнем upload;
  загрузку CPU вкладки-хоста и джиттер доставки; долю соединений, требующих
  TURN (поднять тестовый `coturn`); примерную стоимость релея.
- **Критерий go:** хост держит 8 игроков с запасом (рамка плана).
  **No-go:** пересмотр архитектуры (гибрид / сохранение авторитетного сервера).
  Жёсткую замену НЕ начинать.

**0.2. Spike B — Worker Handoff (сериализация мира Rapier между инстансами)**

- Прототип: `serialize_state()` / `deserialize_state()` мира Rapier + счёта
  (у rapier есть serde-поддержка мира); перенос между двумя WASM-инстансами,
  `worker.terminate()` старого, проверка продолжения симуляции без визуального
  разрыва (≤1–2 c).
- **Критерий go:** бесшовный handoff → реализуем 5.2 «эстафетой».
  **No-go:** fallback в 5.2 (обновление на границе карты + reconnect). Не
  блокирует остальной план.

**0.3. Spike C — Детерминизм WASM (информационный, НЕ блокер)**

- Ожидание: одинаковый WASM-бинарь + идентичный ввод → бит-в-бит идентичный
  результат на любых машинах/браузерах. Спецификация WASM фиксирует семантику
  IEEE 754 для float (кроме NaN-паттернов), `rapier2d` собирается с
  `enhanced-determinism`.
- Прогнать один WASM-core идентичным вводом на 2–3 машинах/браузерах,
  подтвердить ожидание; отдельно замерить реальный источник расхождений —
  тайминг и порядок обработки ввода (разный момент прихода пакетов между
  тиками).
- **Цель:** оценить бюджет коррекции предсказания (`visualError`) и параметры
  сглаживания. Влияет на параметры, не на go/no-go.
- Явно: консенсус-анти-чит на детерминизме **не делаем** (решение «минимум»).

### Этап 1: Инфраструктура лобби (Node.js Мастер-сервер)

**Цель:** Создать Signaling Server (маршрутизатор WebRTC) и реестр серверов.
Никакой игровой логики, только координация соединений. Этап независим от Rust.

**1.1. Инициализация проекта Мастер-сервера**

- `src/master/` в этом репозитории; `src/master/main.js` — новая точка входа
  проекта (заменит `src/server/main.js` на вехе демонтажа).
- Стек: `Node.js` + `Express` (уже в проекте; Fastify не вводим ради
  консистентности) + `ws` (Signaling-сокеты) + ViteExpress для раздачи клиента
  в dev; в prod статику отдаёт Nginx, мастер — только REST + WS.
- Структура данных в памяти: `Map<hostId, HostSession>` (`hostId` — `uuid`,
  уже в зависимостях), где `HostSession` содержит: `name`, `maxPlayers`,
  `currentPlayers`, `mapName`, `region`, `ip`, `status`, `reportCount`.
  Redis — только при реальной необходимости масштаба; для старта не нужен.

**1.2. Реализация REST API для списка серверов**

- Эндпоинт `GET /servers`. Принимает query-параметры: `offset`, `limit`,
  `region`, `search`.
- **Логика:**
  1. Если передан `search`, выполнить поиск по подстроке в поле `name` (без
     учёта регистра), вернуть результаты, игнорируя остальные параметры.
  2. Если общее количество записей в `Map` <= 15, вернуть весь массив.
  3. Иначе применить фильтрацию по `region` и срез по `offset` / `limit`.
- Гео-определение региона — по IP хоста при регистрации: `geoip-lite` либо
  готовый заголовок от Nginx/CDN (решить при реализации — второе бесплатнее
  по памяти).

**1.3. Signaling Server (Маршрутизация WebRTC)**

- WebSocket-хэндлеры:
  - `register_host` — хост сообщает о создании сервера (передаёт конфигурацию).
  - `webrtc_offer` — клиент отправляет SDP-оффер для конкретного `hostId`.
  - `webrtc_answer` — хост отвечает клиенту.
  - `ice_candidate` — обмен ICE-кандидатами для пробития NAT (STUN/TURN).
  - `ping_host` / `pong_host` — микро-сигналы для замера задержки. Замер
    **приблизительный** (путь клиент→мастер→хост, не P2P RTT; честный RTT
    доступен только после ICE-коннекта) — так и подавать в UI.
  - `report_host` — жалоба `/ban` от клиента (см. 5.3).
- Хост держит сигнальный WS постоянно (heartbeat, актуализация
  `currentPlayers`); клиент — на время лобби и установки соединения,
  переоткрывает для `report_host`.
- STUN обязателен; TURN — опциональный релей по итогам Spike A.

**1.4. Защита от DDOS и спама**

- Интегрировать `rate-limiter-flexible` (или аналог).
- Лимиты: не более 1 сервера на 1 IP, ограничение `ping_host` в секунду с
  одного IP-клиента (конкретное значение — настраиваемое).
- Origin-allowlist — переиспользовать паттерн `src/lib/security.js`.

### Этап 2: Единое ядро симуляции (Rust -> WASM)

**Цель:** Перенести симуляцию на Rust и скомпилировать в один `wasm`. Это
снимает проблему производительности моста JS-WASM. Мета остаётся JS (рамка
плана). Порт ведётся **вертикальными срезами**, а не монолитом: сперва один
танк проходит весь путь, затем логика расширяется.

**2.0. Каркас и тулчейн**

- `cargo`-проект в **`core/` в корне репо** (не в `src/`: там JS-исходники
  Vite, `target/` и Rust-код им не место), target `wasm32-unknown-unknown`.
- Зависимости: `rapier2d` (нативная физика, feature `enhanced-determinism`;
  serde-фичи для сериализации мира — нужны Spike B/5.2), `wasm-bindgen`,
  `serde`/`serde_json` для загрузки конфигов. Версии зафиксировать.
- Сборка `wasm-pack` под **два таргета**: `web` (браузер/Worker) и `nodejs`
  (тестовый харнесс под Vitest, п. 2.5).
- Интеграция с Vite: module worker (`new Worker(new URL(...), { type:
  'module' })`), загрузка wasm внутри Worker; npm-скрипты `core:build` и
  включение в `npm run build`.
- Зафиксировать публичный ABI: методы шага симуляции, приёма ввода, сборки
  снапшота, загрузки данных (см. 2.2).

**2.1. Вертикальный MVP-срез — один танк e2e через WASM (без сети)**

- Серверная сторона: `step(dt)`, приём ввода, `pack_body()`/`pack_frame()`
  (v3-раскладка из `src/config/opcodes.js` + `src/lib/snapshotCodec.js`),
  zero-copy через `*const u8`/`*const f32` +
  `new Uint8Array(wasm.memory.buffer, ptr, len)`.
- На клиентской стороне среза кадр читает существующий JS-путь
  (`unpackFrame` → `SnapshotInterpolator`) — клиентская математика не
  трогается до п. 2.6.
- Довести до состояния: «танк ездит через Rust-WASM локально».

**2.2. Перенос симуляции и API ядра**

- Глобальная структура `GameState`: мир Rapier, танки, снаряды, боты.
- Источники для порта:
  - `src/server/parts/` — `Tank`, `BaseModel`, `Map` (физическая часть),
    `Bomb`, `HitscanService`.
  - `src/server/modules/Game.js` — `applyDamage`, урон, таймеры бомб.
- Оружие: hitscan (мгновенный луч) и explosive (физический снаряд) — оба типа.
- `RoundManager` и `ParticipantManager` **не портируются** (мета): ядро
  управляется командами и отчитывается событиями.
- Черновой публичный API (уточняется в 2.0/2.1):
  - команды: `init(config_json)`, `load_map(map_json)`,
    `spawn_tank(id, team, x, y, ...)`, `remove_tank(id)`,
    `apply_input(id, seq, keys)`, `add_bot(id, ...)`, `remove_bot(id)`,
    `step(dt)`;
  - выходы: `take_events()` — буфер событий за тик (kill/death/hit/взрыв —
    топливо для RoundManager/Stat/Panel/звука), `pack_body()` +
    `pack_frame(game_id)` — бинарные кадры v3, `last_input_seq(id)`;
  - для handoff: `serialize_state()` / `deserialize_state(dump)`.

**2.3. Перенос ботов**

- `src/server/modules/bots/` — `BotController`, `NavigationSystem`,
  `Pathfinder`, `SpatialManager` переписываются на Rust: движение ботов —
  часть симуляции, их ввод генерируется внутри ядра на тике.
- `BotManager` (владение контроллерами, связка с реестром) остаётся тонким
  JS-слоем в Worker поверх `add_bot`/`remove_bot`.

**2.4. ABI данных и генерация снапшотов**

- Формат загрузки данных в WASM: `init(config_json)` для `src/data/weapons.js`
  и `src/data/models.js` (+ нужные куски `src/config/game.js`),
  `load_map(map_json)`.
- Карты конвертируются из JS-модулей `src/data/maps/*.js` в статичные `.json`
  — общий шаг с 5.1 (доставка карт хосту без пересборки).
- Упаковка снапшотов (v3 layout) — напрямую в Rust: `pack_frame()` отдаёт
  буфер, готовый к отправке. Распаковка на клиенте остаётся JS
  (`unpackFrame`) до п. 2.6.

**2.5. Тесты ядра**

- Rust `cargo test` для логики + JS↔WASM интеграционный харнесс (nodejs-таргет
  ядра под Vitest).
- **Эталон паритета порта:** пока жив текущий сервер, интеграционный харнесс
  (`tests/server/integration/harness.js`) — источник истины о поведении;
  сценарии прогоняются против ядра и сверяются.
- `TankPredictorParity.test.js` переориентируется: «JS-реплика предикта vs
  Rust-ядро» (вместо «реплика vs Rapier-JS») и живёт до п. 2.6, затем
  удаляется вместе с JS-репликой.
- Судьба текущего 731 теста:
  - **выживают**: мета (Chat/Vote/Stat/Panel/RoundManager/CommandProcessor/
    VoteCoordinator/ParticipantManager), клиентские MVC/view/controller,
    `lib/` (кроме упаковки snapshotCodec);
  - **инвалидируются**: Rapier-parts (`Tank`, `Bomb`, `Map`, `HitscanService`,
    `Game`), упаковка `snapshotCodec` (уезжает в `cargo test` + JS↔WASM
    round-trip), `SocketManager`/`socket`, `RTTManager`;
  - на каждом срезе фиксировать: что заменено Rust-тестом, что — харнессом,
    что удалено.

**2.6. Финальный срез — клиентская математика (выполняется ПОСЛЕ Этапов 3–4)**

- Перенести `SnapshotInterpolator`, `TankPredictor`, `ShotPredictor` и
  `unpackFrame` внутрь ядра: кольцевой буфер кадров +
  `get_render_state(server_time)` (интерполяция) + `apply_local_input(...)`
  (предикт **только локального** танка, минуя сетевой буфер).
- Data-driven рендер-конвейер: `renderTick` вызывает `step_client(dt)`, читает
  плоский `Float32Array` позиций из памяти WASM, применяет к спрайтам
  (`GameCtrl.parse` упрощается до примитивного применения данных).
- Звук и партиклы: WASM складывает события тика в буфер; JS считывает и по
  `eventId` вызывает `SoundManager` / контроллеры эффектов.
- После среза: удалить JS-предикторы, JS-реплику движения и паритет-тест.

### Этап 3: Рефакторинг Клиента (Оболочка на JavaScript)

**Цель:** Пересадить клиент с WebSocket на WebRTC, сохранив существующий
протокол портов. Рендер, звук и клиентская математика на этом этапе не
меняются (см. 2.6).

**3.1. Сетевой слой (WebRTC Data Channels)**

- Удалить игровой `WebSocket` из `src/client/main.js`. Написать `WebRtcManager`
  (нативный `RTCPeerConnection`; `simple-peer` — только если ручной код окажется
  заметно объёмнее).
- Два канала по разделу «Сетевая модель»: `meta` (reliable-ordered) и `state`
  (unreliable-unordered). Диспетчер `socketMethods[portId]` в
  `src/client/main.js` сохраняется как есть — меняется только транспорт.
- `SnapshotInterpolator`: вставка кадров в буфер по `seq` с дедупликацией;
  события опоздавших кадров выдаются немедленно (см. «Сетевая модель»).
- Связь с Мастер-сервером остаётся на WebSocket (только сигналинг).

**3.2. Лобби и Умный Пинг (Lazy Pinging)**

- UI списка серверов: пагинация («Загрузить ещё»), поиск, регионы (Этап 1).
- Внедрить `IntersectionObserver`: когда карточка сервера попадает в видимую
  зону, клиент отправляет `ping_host` через Мастер.
- При получении `pong_host` рассчитать задержку, обновить значение в DOM и
  применить пересортировку. В UI подавать как приблизительную оценку (см. 1.3).

**3.3. Обработка разрывов**

- `dataChannel.onclose`/`connectionstatechange` → остановка рендера
  (`Application.stop()`), UI-заглушка, редирект в Лобби.
- Учесть: выход хоста = смерть комнаты (host-migration нет) — честный UI.

### Этап 4: Разработка Браузерного Хоста (Web Worker)

**Цель:** Развернуть серверную часть игры на клиенте инициатора, изолировав её
в отдельном потоке.

**4.1. Архитектура Главного Потока (Main Thread)**

- **Критическое правило:** `RTCPeerConnection` (WebRTC) **создаются и живут в
  Главном Потоке браузера хоста**, а не внутри Worker'а.
- Главный поток — роутер: ловит бинарные пакеты по WebRTC и пересылает в Worker
  через `worker.postMessage()` (с Transferable `ArrayBuffer`, чтобы избежать
  копий), и обратно. Это же позволяет убивать Worker без разрыва
  P2P-соединений (основа Worker Handoff).

**4.2. Настройка Web Worker (`host.worker.js`)**

- Worker загружает WASM-ядро и JS-мету (карта модулей выше): фасад,
  RoundManager, ParticipantManager, Chat/Vote/Stat/Panel, TimerManager,
  CommandProcessor, VoteCoordinator, auth-валидация.
- Игровой цикл внутри Worker'а (~120 Гц): таймеры Worker'а не троттлятся
  браузером в фоновой вкладке (в отличие от главного потока) — это ключевая
  причина изоляции, помимо отзывчивости UI.
- На каждый тик Worker вызывает `step(dt)`, забирает события (`take_events()`)
  для меты и готовые бинарные кадры (`pack_frame`) для каждого `gameId`,
  передаёт в Главный поток (`postMessage` + Transferable), который рассылает
  их по DataChannel согласно сетевой модели (`meta`/`state`).
- **Бэкпрешер:** контроль `bufferedAmount`/`bufferedAmountLowThreshold` на
  каждом DataChannel; медленным пирам — пропуск state-кадров (перенос
  per-peer-логики `SnapshotManager` на новый транспорт).
- **Жизненный цикл подключения клиента** сохраняется поверх `meta`-канала:
  порты 0→4 (config → auth → map → first shot), затем игровой цикл. Auth
  (валидация имени/модели, уникальность) выполняет Worker
  (`src/lib/validators.js`, `ParticipantManager.checkName`).

**4.3. Инициализация комнаты**

- Имя/карта/лимит игроков (≤ 8, рамка плана)/таймеры, введённые пользователем
  на сайте, передаются в Worker при старте для конфигурации ядра и меты;
  хост регистрируется у Мастера (`register_host`).

**4.4. Хост-игрок (loopback)**

- Создатель комнаты играет в той же вкладке: его клиентская часть (обычный
  клиентский код) обменивается с Worker напрямую через `postMessage`-loopback
  — те же сообщения портов и кадры, но без WebRTC. Для клиентского кода
  транспорт прозрачен (общий интерфейс с `WebRtcManager`).

**4.5. Разделение ролей (опционально, UX)**

- Вкладка «Создать сервер» может показывать только консоль/панель управления.
- При анти-чите «минимум» это чисто удобство, НЕ мера защиты (защитная функция
  «второй вкладки» от читов хоста отпадает).

### Этап 5: Бесшовные обновления и соц-модерация

**Цель:** Автоматическая доставка кода и данных без кика игроков + минимальная
модерация. Тяжёлый анти-чит вырезан (решение «минимум»).

**5.1. Динамическая загрузка данных (Карты)**

- Карты (`src/data/maps/*.js`) конвертируются в статичные `.json` на
  Мастер-сервере (формат совпадает с `load_map` ядра, п. 2.4).
- Мастер шлёт `update_available: map_name`.
- Главный поток хоста делает `fetch()`, скачивает JSON и передаёт в Worker.
- Worker вызывает `load_map(json)` — карта готова к следующему раунду.

**5.2. Обновление WASM-кода — по результату Spike B**

- **Если Handoff подтверждён (go):** по сигналу `new_wasm_version` Главный поток
  в фоне скачивает новый `.wasm`; в конце раунда отправляет Worker'у `SLEEP`;
  Worker сериализует состояние (`serialize_state()` → `Uint8Array`); Главный
  поток `terminate()` старого Worker'а, создаёт нового, передаёт дамп, новый
  WASM делает `deserialize_state()`; пересылка WebRTC-пакетов возобновляется
  (для клиентов — лаг 1–2 c между раундами).
- **Если Handoff отклонён (no-go):** fallback — обновление на границе карты с
  коротким reconnect (проще, приемлемо для казуальной игры).

**5.3. Социальная модерация (`/ban`) — единственная анти-чит-мера**

- Команда `/ban <причина>` (причина обязательна, публично не отображается)
  перехватывается **на клиенте** — до отправки хосту: серверный
  `CommandProcessor` исполняется у хоста, и жалоба, идущая через него, могла бы
  фильтроваться читером. Клиентский перехват — в контроллере чата перед
  отправкой на порт CHAT.
- Жалоба уходит напрямую Мастеру по сигнальному WS (`report_host`:
  `hostId` + `reporterId`; переоткрыть WS при необходимости). Уникальность
  репортеров — чтобы один игрок не накрутил жалобы.
- Логика Мастера: при `X` уникальных жалобах за час (напр. 5 при лимите 8
  игроков) `status` → `banned`.
- Мастер прекращает отдавать сервер в `GET /servers` и рвёт сигнальный WS.
  Читер остаётся в комнате один, новые игроки к нему не попадают.

**5.4. Гигиена среды (НЕ анти-чит)**

- **Strict CORS & Origin:** Мастер отклоняет WS-соединения с чужим `Origin`
  (переиспользовать паттерн `src/lib/security.js`).
- **CSP:** браузер грузит `.wasm` только с официального источника.
- **Обфускация:** минифицировать/обфусцировать JS-оболочку.
- Явно: это отсекает «уличных» злоумышленников, но НЕ хоста-читера — он владеет
  своим процессом и WASM-памятью. Подробнее — [P2P-PLAN-SAFITY.md](P2P-PLAN-SAFITY.md).
