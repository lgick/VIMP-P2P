# Развернутый план миграции VIMP Tank Battle (P2P + Rust WASM)

### Этап 1: Инфраструктура лобби (Node.js Мастер-сервер)

**Цель:** Создать Signaling Server (маршрутизатор WebRTC) и реестр серверов.
Никакой игровой логики, только координация соединений.

**1.1. Инициализация проекта Мастер-сервера**

- Создать отдельную директорию/проект (например, `src/master/`).
- Стек: `Node.js` + `Fastify` или `Express` (для REST API) + `ws` (для
  Signaling-сокетов).
- Структура данных в памяти (или в Redis): `Map<HostId, HostSession>`, где
  `HostSession` содержит: `name`, `maxPlayers`, `currentPlayers`, `mapName`,
  `region`, `ip`, `status`, `reportCount`.

**1.2. Реализация REST API для списка серверов**

- Эндпоинт `GET /servers`. Принимает query-параметры: `offset`, `limit`,
  `region`, `search`.
- **Логика:**
  1. Если передан `search`, выполнить поиск по подстроке в поле `name` (без
     учета регистра), вернуть результаты, игнорируя остальные параметры.
  2. Если общее количество записей в `Map` <= 15, вернуть весь массив.
  3. Иначе применить фильтрацию по `region` (гео-определение по IP хоста при
     регистрации) и срез по `offset` / `limit`.

**1.3. Signaling Server (Маршрутизация WebRTC)**

- WebSocket-хэндлеры:
  - `register_host` — хост сообщает о создании сервера (передает конфигурацию).
  - `webrtc_offer` — клиент отправляет SDP-оффер для конкретного `HostId`.
  - `webrtc_answer` — хост отвечает клиенту.
  - `ice_candidate` — обмен ICE-кандидатами для пробития NAT (STUN/TURN).
  - `ping_host` / `pong_host` — микро-сигналы для замера задержки.

**1.4. Защита от DDOS и спама**

- Интегрировать `rate-limiter-flexible` (или аналог).
- Лимиты: не более 1 сервера на 1 IP, не более 50 `ping_host` запросов в секунду
  с одного IP-клиента.

### Этап 2: Единое ядро симуляции (Rust -> WASM)

**Цель:** Написать всю физику и логику на Rust, чтобы скомпилировать её в один
`wasm` файл. Это решает проблему производительности моста JS-WASM.

**2.1. Инициализация Rust-проекта**

- Создать `cargo` проект (внутри `src/core/`), target: `wasm32-unknown-unknown`.
- Зависимости: `rapier2d` (нативная физика, без JS-оберток), `wasm-bindgen` (для
  моста API), `serde` (для сериализации состояний), `bincode` (для упаковки
  бинарного снапшота).

**2.2. Перенос состояния (State Management)**

- Создать глобальную структуру `GameState`, хранящую: `world: rapier2d::World`,
  `players: HashMap<u32, Player>`, `projectiles: Vec<Projectile>`,
  `round_state: RoundState`.
- Перенести логику `RoundManager`, `Game.applyDamage`, `HitscanService` из JS в
  соответствующие методы `GameState`.

**2.3. Перенос механизмов сглаживания (Prediction & Interpolation)**

- Внедрить клиентскую математику внутрь Rust.
- На сервере: Rust выполняет `step()` и возвращает бинарный дамп.
- На клиенте: Тот же самый `wasm` получает дамп, сохраняет его в кольцевой буфер
  и делает метод `get_render_state(server_time)`. Rust сам интерполирует позиции
  (`lerp`) и отдает плоский массив `[id, x, y, angle]`.
- Prediction: Метод `apply_local_input(input)` применяет физику только к
  локальному танку, минуя сетевой буфер.

**2.4. Прямая работа с памятью (Zero-copy API)**

- Вместо сериализации данных в JSON, экспортировать методы, возвращающие
  указатели на память (`*const f32`).
- Из JS читать эти данные через
  `new Float32Array(wasm.memory.buffer, pointer, length)`.
- Упаковка снапшотов (v3 layout из `snapshotCodec.js`) реализуется напрямую в
  Rust: метод `build_snapshot_frame()` отдает `Uint8Array`, готовый к отправке
  по сети.

### Этап 3: Рефакторинг Клиента (Оболочка на JavaScript)

**Цель:** Очистить клиентский JS (`src/client/`) от сложной математики. JS
только ловит ввод, отправляет его по WebRTC и рисует данные из WASM.

**3.1. Сетевой слой (WebRTC Data Channels)**

- Удалить инстанс `WebSocket` для игры. Написать `WebRtcManager` поверх
  `simple-peer` (или нативного `RTCPeerConnection`).
- Связь с Мастер-сервером остается на WebSocket (только для сигналов). Игровой
  трафик (порт 5) идет через WebRTC `RTCDataChannel`.
- Реализация обработки разрывов: `dataChannel.onclose` вызывает
  `Application.stop()` и UI заглушку с редиректом.

**3.2. Лобби и Умный Пинг (Lazy Pinging)**

- Внедрить `IntersectionObserver` в UI списка серверов.
- Когда карточка сервера попадает в видимую зону экрана, клиент отправляет
  `ping_host` через Мастер-сервер.
- При получении `pong_host` рассчитывать RTT, обновлять значение в DOM и
  применять пересортировку массива.

**3.3. Рендер-конвейер и Звук (Data-Driven)**

- Изменить метод `renderTick`:
  1. JS вызывает `wasm.step_client(dt)`.
  2. JS читает `Float32Array` с позициями сущностей.
  3. Вызов `sprite.position.set(x, y)`.
- События (`AudioEvent`, создание/удаление партиклов): WASM складывает события
  за тик в отдельный буфер. JS считывает буфер, если видит
  `eventId == EXPLOSION`, вызывает `SoundManager.play(...)`.

### Этап 4: Разработка Браузерного Хоста (Web Worker)

**Цель:** Развернуть серверную логику на клиенте инициатора, полностью
изолировав её в отдельном потоке.

**4.1. Архитектура Главного Потока (Main Thread)**

- **Критическое правило:** `RTCPeerConnection` (WebRTC) **должны создаваться и
  жить в Главном Потоке браузера хоста**, а не внутри Worker'а.
- Главный поток выполняет роль роутера: ловит бинарные пакеты по WebRTC и
  пересылает их в Worker через `worker.postMessage()`. Это позволит в будущем
  убивать Worker без разрыва P2P-соединений.

**4.2. Настройка Web Worker (`host.worker.js`)**

- Worker загружает `vimp_core_bg.wasm`.
- Реализовать игровой цикл внутри Worker'а через `setInterval(fn, 1000 / 120)`
  или `setTimeout`.
- На каждый тик Worker вызывает `wasm.step_server()`, забирает массив готовых
  бинарных снапшотов (кадров) для каждого `gameId` и передает в Главный поток
  (`postMessage`), который рассылает их по DataChannels.

**4.3. Разделение Хоста и Клиента**

- Вкладка "Создать сервер" после создания показывает только консоль (логи:
  «Игрок Х зашел»).
- Для игры хост открывает новую вкладку, заходит на свой Мастер-сервер, находит
  свой сервер в Лобби и подключается по WebRTC (браузер автоматически
  оптимизирует это до loopback-соединения). Это решает проблему читов через
  консоль хоста-игрока.

### Этап 5: Бесшовные обновления и Анти-чит

**Цель:** Автоматическая доставка кода и данных без кика игроков + защита
честной игры.

**5.1. Динамическая загрузка данных (Карты)**

- Карты (`src/data/maps/`) перенести в статичные `.json` файлы на
  Мастер-сервере.
- Мастер-сервер шлет команду `update_available: map_name`.
- Главный поток хоста делает `fetch()`, скачивает JSON и передает в Worker
  (`postMessage`).
- Worker вызывает `wasm.load_map(json)` — карта готова к старту следующего
  раунда.

**5.2. Worker Handoff (Обновление WASM-кода)**

- По сигналу `new_wasm_version` от Мастер-сервера Главный поток хоста в фоне
  скачивает новый `.wasm` файл.
- Дождавшись конца текущего раунда, Главный поток отправляет Worker'у команду
  `SLEEP`.
- Worker вызывает внутри WASM метод сериализации `serialize_state()`, который
  сбрасывает мир Rapier и счет игроков в бинарный `Uint8Array`.
- Worker передает массив в Главный поток. Главный поток делает
  `worker.terminate()`.
- Главный поток создает `new Worker()`, инициализирует новый WASM, передает туда
  массив. Новый WASM делает `deserialize_state()`.
- Главный поток возобновляет пересылку WebRTC-пакетов в новый Worker. Игра
  продолжается (для клиентов это выглядит как лаг в 1-2 секунды между раундами).
