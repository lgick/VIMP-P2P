Детализированный пошаговый план миграции VIMP Tank Battle на P2P-архитектуру с
Rust-ядром.

**Этот план составлен как исчерпывающее техническое задание, но не является
конечным решением, поэтому не содержит обязательных рекомендаций!**

**Важно: проект находится в открытом доступе на github, нужно учитывать это в
выборе решений по безопасности и читингу.**

**Рамки (согласованы):** анти-чит — минимум (`/ban` + гигиена среды; хост может
читерить — принято); миграция — жёсткая замена, без параллельного режима; сперва
Этап 0 (снятие рисков) — только потом демонтаж рабочего сервера. Обоснование —
в [P2P-PLAN.md](P2P-PLAN.md), модель угроз — в
[P2P-PLAN-SAFITY.md](P2P-PLAN-SAFITY.md).

# Развернутый план миграции VIMP Tank Battle (P2P + Rust WASM)

### Этап 0: Снятие киллер-рисков (go/no-go перед жёсткой заменой)

**Цель:** Дешёвыми PoC проверить допущения, на которых держится архитектура, ДО
демонтажа рабочего сервера. Пока главный спайк не пройден — код не трогаем.

**0.1. Spike A — Пропускная способность WebRTC + upload хоста (главный go/no-go)**

- Замерить фактический размер бинарного кадра (`SnapshotPacker.packBody`,
  `src/lib/snapshotCodec.js`) для типичной сцены (N танков + снаряды + карта).
- Прототип: один браузер-хост шлёт синтетические кадры этого размера через
  `RTCDataChannel` (unreliable/unordered) на N подключений на 30 Гц.
- Замерить деградацию при 4/8/12 игроках на типичном домашнем upload
  (~10 Мбит/с); долю соединений, требующих TURN (поднять тестовый `coturn`);
  примерную стоимость релея.
- **Критерий go:** хост держит целевое число игроков (напр. 8) с запасом.
  **No-go:** пересмотр архитектуры (гибрид / сохранение авторитетного сервера).
  Жёсткую замену НЕ начинать.

**0.2. Spike B — Worker Handoff (сериализация мира Rapier между инстансами)**

- Прототип: `serialize_state()` / `deserialize_state()` мира Rapier + счёта;
  перенос между двумя WASM-инстансами, `worker.terminate()` старого, проверка
  продолжения симуляции без визуального разрыва (≤1–2 c).
- **Критерий go:** бесшовный handoff → реализуем 5.2 «эстафетой».
  **No-go:** fallback в 5.2 (обновление на границе карты + reconnect). Не
  блокирует остальной план.

**0.3. Spike C — Дрейф детерминизма (информационный, НЕ блокер)**

- Один и тот же WASM-core прогнать идентичным вводом на 2–3 машинах/браузерах,
  замерить расхождение позиций за N секунд.
- **Цель:** оценить бюджет коррекции предсказания (`visualError`) и устойчивость
  реплей-реконсилиации на клиенте. Влияет на параметры сглаживания, не на
  go/no-go.
- Явно: консенсус-анти-чит на детерминизме **не делаем** (решение «минимум»).

### Этап 1: Инфраструктура лобби (Node.js Мастер-сервер)

**Цель:** Создать Signaling Server (маршрутизатор WebRTC) и реестр серверов.
Никакой игровой логики, только координация соединений. Этап независим от Rust.

**1.1. Инициализация проекта Мастер-сервера**

- Создать отдельную директорию/проект (например, `src/master/`).
- Стек: `Node.js` + `Fastify` или `Express` (для REST API) + `ws` (для
  Signaling-сокетов).
- Структура данных в памяти: `Map<HostId, HostSession>`, где `HostSession`
  содержит: `name`, `maxPlayers`, `currentPlayers`, `mapName`, `region`, `ip`,
  `status`, `reportCount`. Redis — только при реальной необходимости масштаба;
  для старта не нужен.

**1.2. Реализация REST API для списка серверов**

- Эндпоинт `GET /servers`. Принимает query-параметры: `offset`, `limit`,
  `region`, `search`.
- **Логика:**
  1. Если передан `search`, выполнить поиск по подстроке в поле `name` (без
     учёта регистра), вернуть результаты, игнорируя остальные параметры.
  2. Если общее количество записей в `Map` <= 15, вернуть весь массив.
  3. Иначе применить фильтрацию по `region` (гео-определение по IP хоста при
     регистрации) и срез по `offset` / `limit`.

**1.3. Signaling Server (Маршрутизация WebRTC)**

- WebSocket-хэндлеры:
  - `register_host` — хост сообщает о создании сервера (передаёт конфигурацию).
  - `webrtc_offer` — клиент отправляет SDP-оффер для конкретного `HostId`.
  - `webrtc_answer` — хост отвечает клиенту.
  - `ice_candidate` — обмен ICE-кандидатами для пробития NAT (STUN/TURN).
  - `ping_host` / `pong_host` — микро-сигналы для замера задержки.
- STUN обязателен; TURN — опциональный релей (нужен по итогам Spike A для
  части соединений, недоступных напрямую).

**1.4. Защита от DDOS и спама**

- Интегрировать `rate-limiter-flexible` (или аналог).
- Лимиты: не более 1 сервера на 1 IP, ограничение `ping_host` в секунду с
  одного IP-клиента (конкретное значение — настраиваемое).
- Origin-allowlist — переиспользовать паттерн `src/lib/security.js`.

### Этап 2: Единое ядро симуляции (Rust -> WASM)

**Цель:** Написать всю физику и логику на Rust и скомпилировать в один `wasm`.
Это снимает проблему производительности моста JS-WASM. Порт ведётся
**вертикальными срезами**, а не монолитом: сперва один танк проходит весь путь,
затем логика расширяется.

**2.0. Каркас**

- Создать `cargo` проект (внутри `src/core/`), target: `wasm32-unknown-unknown`.
- Зависимости: `rapier2d` (нативная физика, без JS-оберток, feature
  `enhanced-determinism`), `wasm-bindgen` (мост API), `serde` + сериализатор для
  упаковки бинарного снапшота.
- Зафиксировать публичный ABI: методы шага симуляции, приёма ввода, сборки
  снапшота, загрузки данных.

**2.1. Вертикальный MVP-срез — один танк e2e через WASM (без сети)**

- Серверная сторона: `step_server(dt)`, приём ввода, `build_snapshot_frame()`
  (v3-раскладка из `src/config/opcodes.js` + `src/lib/snapshotCodec.js`),
  zero-copy через `*const f32` + `new Float32Array(wasm.memory.buffer, ptr, len)`.
- Клиентская сторона того же WASM: кольцевой буфер + `get_render_state(server_time)`
  (интерполяция) + `apply_local_input(input)` (предикт **только локального**
  танка, минуя сетевой буфер).
- Довести до состояния: «танк ездит через Rust-WASM локально».

**2.2. Перенос полной логики (State Management)**

- Глобальная структура `GameState`: `world: rapier2d::World`,
  `players: HashMap<u32, Player>`, `projectiles: Vec<Projectile>`,
  `round_state: RoundState`.
- Источники для порта:
  - `src/server/parts/` — `Tank`, `BaseModel`, `Map`, `Bomb`, `HitscanService`.
  - `src/server/modules/Game.js` — `applyDamage`, урон, таймеры бомб.
  - `src/server/core/RoundManager.js` — раунды/команды/карты.
  - `src/server/player/ParticipantManager.js` — реестр участников.
- Оружие: hitscan (мгновенный луч) и explosive (физический снаряд) — перенести
  оба типа.

**2.3. Перенос ботов**

- `src/server/modules/bots/` — `BotController`, `NavigationSystem`, `Pathfinder`,
  `SpatialManager`. Навигация/поиск пути/пространственный хэш переписываются на
  Rust.

**2.4. ABI данных и генерация снапшотов**

- Определить формат загрузки данных в WASM: `load_map(json)`,
  `load_config(...)` для `src/data/weapons.js` и `src/data/models.js`.
- Карты → JSON (совпадает с 5.1), чтобы их можно было доставлять хосту без
  пересборки.
- Упаковка снапшотов (v3 layout) — напрямую в Rust: `build_snapshot_frame()`
  отдаёт `Uint8Array`, готовый к отправке. `unpackFrame` — тоже в Rust (клиент).

**2.5. Тесты ядра**

- Rust `cargo test` для логики + JS↔WASM интеграционный харнесс.
- Явно зафиксировать: парити-тест `TankPredictorParity` теряет смысл (одно ядро
  вместо двух реализаций движения); ~730 текущих тестов частично инвалидируются
  — составить план их замены/переноса (что покрывается Rust-тестами, что —
  харнессом, что удаляется).

### Этап 3: Рефакторинг Клиента (Оболочка на JavaScript)

**Цель:** Очистить клиентский JS (`src/client/`) от сложной математики. JS
только ловит ввод, отправляет его по WebRTC и рисует данные из WASM.

**3.1. Сетевой слой (WebRTC Data Channels)**

- Удалить игровой `WebSocket` из `src/client/main.js`. Написать `WebRtcManager`
  поверх `simple-peer` (или нативного `RTCPeerConnection`).
- Каналы данных: **unreliable-unordered** для снапшотов (порт 5, аналог UDP) и
  **reliable-ordered** для меты (panel/stat/chat/vote/keyset).
- Связь с Мастер-сервером остаётся на WebSocket (только сигналинг).
- Обработка разрывов: `dataChannel.onclose` вызывает `Application.stop()` и
  UI-заглушку с редиректом. Учесть: выход хоста = смерть комнаты (host-migration
  нет) — честный UI.

**3.2. Лобби и Умный Пинг (Lazy Pinging)**

- Внедрить `IntersectionObserver` в UI списка серверов.
- Когда карточка сервера попадает в видимую зону, клиент отправляет `ping_host`
  через Мастер-сервер.
- При получении `pong_host` рассчитать RTT, обновить значение в DOM и применить
  пересортировку.

**3.3. Рендер-конвейер и Звук (Data-Driven)**

- Изменить `renderTick`:
  1. JS вызывает `wasm.step_client(dt)`.
  2. JS читает `Float32Array` с позициями сущностей.
  3. Вызов `sprite.position.set(x, y)`.
- События (`AudioEvent`, создание/удаление партиклов): WASM складывает события
  за тик в отдельный буфер. JS считывает буфер и по `eventId` (например,
  `EXPLOSION`) вызывает `SoundManager.play(...)`.

### Этап 4: Разработка Браузерного Хоста (Web Worker)

**Цель:** Развернуть серверную логику на клиенте инициатора, изолировав её в
отдельном потоке.

**4.1. Архитектура Главного Потока (Main Thread)**

- **Критическое правило:** `RTCPeerConnection` (WebRTC) **создаются и живут в
  Главном Потоке браузера хоста**, а не внутри Worker'а.
- Главный поток — роутер: ловит бинарные пакеты по WebRTC и пересылает в Worker
  через `worker.postMessage()` (по возможности с Transferable `ArrayBuffer`,
  чтобы избежать копий). Это же позволяет убивать Worker без разрыва
  P2P-соединений (основа handoff).

**4.2. Настройка Web Worker (`host.worker.js`)**

- Worker загружает `vimp_core_bg.wasm`.
- Игровой цикл внутри Worker'а (~120 Гц).
- На каждый тик Worker вызывает `wasm.step_server()`, забирает массив готовых
  бинарных кадров для каждого `gameId` и передаёт в Главный поток
  (`postMessage`), который рассылает их по DataChannels.

**4.3. Инициализация комнаты**

- Имя/карта/лимит игроков/таймеры, введённые пользователем на сайте,
  передаются в Worker при старте для конфигурации WASM-модуля.

**4.4. Разделение ролей (опционально, UX)**

- Вкладка «Создать сервер» может показывать только консоль/панель управления.
- При анти-чите «минимум» это чисто удобство, НЕ мера защиты (защитная функция
  «второй вкладки» от читов хоста отпадает).

### Этап 5: Бесшовные обновления и соц-модерация

**Цель:** Автоматическая доставка кода и данных без кика игроков + минимальная
модерация. Тяжёлый анти-чит вырезан (решение «минимум»).

**5.1. Динамическая загрузка данных (Карты)**

- Карты (`src/data/maps/`) перенести в статичные `.json` на Мастер-сервере.
- Мастер шлёт `update_available: map_name`.
- Главный поток хоста делает `fetch()`, скачивает JSON и передаёт в Worker.
- Worker вызывает `wasm.load_map(json)` — карта готова к следующему раунду.

**5.2. Обновление WASM-кода — по результату Spike B**

- **Если Handoff подтверждён (go):** по сигналу `new_wasm_version` Главный поток
  в фоне скачивает новый `.wasm`; в конце раунда отправляет Worker'у `SLEEP`;
  Worker сериализует состояние (`serialize_state()` → `Uint8Array`); Главный
  поток `terminate()` старого Worker'а, создаёт нового, передаёт дамп, новый
  WASM делает `deserialize_state()`; пересылка WebRTC-пакетов возобновляется
  (для клиентов — лаг 1–2 c между раундами).
- **Если Handoff отклонён (no-go):** fallback — обновление на границе карты с
  коротким reconnect (проще, приемлемо для казуальной игры).

**5.3. Социальная модерация (`/ban`) — единственная анти-чит-мера**

- В `Chat.js` добавить команду `/ban` с обязательной причиной (публично не
  отображается).
- Команда уходит на Мастер-сервер с `host_id` + `reporter_id` (чтобы один игрок
  не накрутил жалобы).
- Логика Мастера: при `X` уникальных жалобах за час (напр. 5 при лимите 10
  игроков) `status` → `banned`.
- Мастер прекращает отдавать сервер в `GET /servers` и рвёт сигнальный WS.
  Читер остаётся в комнате один, новые игроки к нему не попадают.

**5.4. Гигиена среды (НЕ анти-чит)**

- **Strict CORS & Origin:** Мастер отклоняет WS-соединения с чужим `Origin`.
- **CSP:** браузер грузит `.wasm` только с официального CDN.
- **Обфускация:** минифицировать/обфусцировать JS-оболочку.
- Явно: это отсекает «уличных» злоумышленников, но НЕ хоста-читера — он владеет
  своим процессом и WASM-памятью. Подробнее — [P2P-PLAN-SAFITY.md](P2P-PLAN-SAFITY.md).
