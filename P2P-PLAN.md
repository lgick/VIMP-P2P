Пошаговый план миграции VIMP Tank Battle на P2P-архитектуру с использованием
единого Rust-ядра (WASM).

План описывает **что** должно быть сделано, оставляя детали реализации на этап
разработки (детальный план в файле
[Детальный план миграции на P2P-архитектуру](P2P-PLAN-DETAILS.md)) .

Важно: проект находится в открытом доступе на github, нужно учитывать это в
выборе решений по безопасности и читинга.

### Этап 1: Инфраструктура лобби (Node.js Мастер-сервер)

Цель: Создать центральный узел для маршрутизации соединений и отображения списка
серверов без обработки игровой логики.

1. **Создание Мастер-сервера**: Разработать легковесный Node.js сервер
   (Signaling Server), хранящий реестр активных комнат (браузерных хостов).
2. **API списка серверов**: Реализовать отдачу списка серверов с поддержкой:
   - Пагинации (для кнопки «Загрузить ещё»).
   - Фильтрации по регионам (по умолчанию).
   - Отключения регионального фильтра, если общее количество серверов ниже
     порогового значения (например, <= 15).
   - Прямого поиска по имени сервера (игнорируя регионы и пагинацию).
3. **Защита от DDOS пингами**: Внедрить жесткое ограничение (Rate Limiting) на
   Мастер-сервере для количества сигнальных ping-запросов от одного IP.
4. **Маршрутизация WebRTC**: Настроить пересылку SDP-офферов и ответов между
   клиентами и браузерными хостами для установки прямого P2P-соединения.

### Этап 2: Единое ядро симуляции (Rust -> WASM)

Цель: Перенести всю математику, физику и правила игры в изолированный,
высокопроизводительный «чёрный ящик», который будет работать как на сервере
(хосте), так и на клиенте.

1. **Интеграция физики**: Инициализировать физический движок (Rapier 2D) нативно
   внутри Rust-проекта (избавившись от JS-обертки `@dimforge/rapier2d-compat`).
2. **Перенос серверной логики**: Переписать модули `Game`, `RoundManager`,
   логику оружия (урон, hitscan, таймеры бомб), систему сущностей
   (`ParticipantManager`) и ИИ ботов на Rust.
3. **Перенос клиентской математики**: Перенести механизмы сглаживания сети
   (`SnapshotInterpolator`, `TankPredictor`, `ShotPredictor`) внутрь Rust-ядра.
4. **Генерация снапшотов**: Перенести логику `SnapshotPacker` (упаковку бинарных
   кадров) и `unpackFrame` в Rust для прямой работы с памятью WASM.
5. **Сборка универсального модуля**: Скомпилировать получившийся Rust-код в
   единый `.wasm` файл, предоставляющий публичные методы (API) для передачи
   ввода и получения готовых координат/кадров.

### Этап 3: Рефакторинг Клиента (Оболочка на JavaScript)

Цель: Превратить текущий клиент (PixiJS) в «глупый» слой ввода-вывода (I/O),
который только отрисовывает данные из WASM и отправляет нажатия клавиш.

1. **Абстракция сетевого слоя**: Заменить использование `ws` (WebSocket) на
   WebRTC Data Channels. Интегрировать логику «Умного пинга» (клиент пингует
   только те серверы, которые в данный момент видны на экране).
2. **Интеграция WASM-модуля**: Настроить передачу пользовательского ввода
   (`InputListener`) напрямую в WASM-модуль.
3. **Связка с рендером**: Изменить конвейер рендеринга (`GameCtrl.parse`).
   Вместо интерполяции внутри JS, PixiJS должен запрашивать у WASM готовый
   плоский массив координат и применять их к спрайтам.
4. **Связка со звуком**: Настроить вызов пространственных звуков (`Howler`) по
   триггерам (событиям), которые отдает WASM-модуль (например, при взрыве
   `w2e`).
5. **Обработка разрывов**: Реализовать UI-перехват закрытия WebRTC-канала с
   остановкой рендера и редиректом пользователя в Лобби.

### Этап 4: Разработка Браузерного Хоста (Web Worker)

Цель: Запуск серверной части игры в фоновом потоке браузера создателя комнаты.

1. **Изоляция в Worker**: Разместить загрузку WASM-ядра и сетевой слой (рассылку
   кадров по WebRTC) в Web Worker, чтобы физический цикл (~120 Гц) не блокировал
   вкладку браузера.
2. **Инициализация настроек**: Настроить передачу параметров (имя, карта, лимит
   игроков, таймеры), введенных пользователем на сайте, в Worker при его запуске
   для конфигурации WASM-модуля.
3. **Разделение ролей (Опционально)**: Реализовать логику, при которой вкладка,
   создавшая сервер, служит только панелью управления. Для игры хост должен
   подключиться к себе как обычный клиент.

### Этап 5: Бесшовные обновления и Анти-чит

Цель: Обеспечить автообновление серверов без потери игроков и минимизировать
влияние недобросовестных хостов.

1. **Динамическая загрузка карт (Обновление данных)**: Удалить жесткую привязку
   карт к сборке. Настроить Worker на скачивание актуальных JSON-карт с
   Мастер-сервера по сигналу `update_available` (для применения в следующем
   раунде).
2. **Эстафетная передача (Обновление кода)**: Реализовать механизм "Worker
   Handoff" при выходе новой версии логики:
   - Остановка цикла в конце раунда.
   - Выгрузка состояния (метаданные + дамп мира Rapier) из старого WASM-модуля.
   - Уничтожение старого Worker'а и запуск нового (с обновленным WASM).
   - Загрузка дампа в новый модуль и продолжение игры без разрыва
     WebRTC-соединений.
3. **Социальная модерация (Анти-чит)**: Внедрить команду в чате `/ban` для
   жалобы на сервер. Настроить Мастер-сервер на исключение хоста из поисковой
   выдачи при достижении лимита жалоб.
4. **Обфускация JS-слоя**: Внедрить жесткую минификацию и обфускацию для
   `server-bundle.js` в процессе CI/CD сборки.
