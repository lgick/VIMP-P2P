Пошаговый план миграции VIMP Tank Battle на P2P-архитектуру с использованием
единого Rust-ядра (WASM).

План описывает **что** должно быть сделано, оставляя детали реализации на этап
разработки (детальный план в файле
[Детальный план миграции на P2P-архитектуру](P2P-PLAN-DETAILS.md)).

Важно: проект находится в открытом доступе на github, нужно учитывать это в
выборе решений по безопасности и читингу.

### Принятые решения (рамки плана)

Эти решения определяют структуру всего плана; отступать от них без пересмотра
плана нельзя.

1. **Анти-чит — минимум.** P2P-хост физически исполняет симуляцию у себя, и
   WASM-память доступна ему из JS-оболочки — хост с модифицированным клиентом
   может читерить, и обфускация/подписи этого не предотвращают. Мы открыто
   принимаем это (казуальная open-source игра) и ограничиваемся соц-модерацией
   (`/ban`, закрытие комнат) и базовой гигиеной среды. Тяжёлые схемы
   (кросс-валидация состояний, реплей-проверки, криптоподписи) рассмотрены и
   отклонены — см. [Модель угроз и анти-чит](P2P-PLAN-SAFITY.md).
2. **Жёсткая замена.** Текущий авторитетный WS-сервер демонтируется; проект
   мигрирует на P2P целиком, без параллельного режима. Именно поэтому Этап 0
   (снятие киллер-рисков) обязателен ДО демонтажа рабочего сервера.
3. **Сначала риски.** Этап 0 — дешёвые PoC с явными go/no-go-критериями. Пока
   главный спайк не пройден, рабочий сервер не трогаем.

### Этап 0: Снятие киллер-рисков (go/no-go перед жёсткой заменой)

Цель: дешёвыми прототипами проверить допущения, на которых держится вся
архитектура, ДО демонтажа рабочего сервера.

1. **Пропускная способность хоста (главный go/no-go)**: проверить, что браузер
   на типичном домашнем upload-канале способен рассылать бинарные снапшоты
   целевому числу игроков на 30 Гц через `RTCDataChannel`. Оценить долю
   соединений, требующих TURN-релея, и его стоимость. No-go → пересмотр
   архитектуры (гибрид / сохранение авторитетного сервера), жёсткую замену не
   начинать.
2. **Worker Handoff**: проверить, что состояние мира Rapier сериализуется и
   переносится между WASM-инстансами без визуального разрыва. Результат
   определяет реализацию Этапа 5 (бесшовное обновление кода или fallback).
3. **Дрейф детерминизма (информационный)**: замерить расхождение симуляции на
   разных машинах при идентичном вводе — чтобы оценить бюджет коррекции
   клиентского предсказания. Не блокер (консенсус-анти-чит не делаем).

### Этап 1: Инфраструктура лобби (Node.js Мастер-сервер)

Цель: Создать центральный узел для маршрутизации соединений и отображения списка
серверов без обработки игровой логики. Этап независим от Rust и низкорисков.

1. **Создание Мастер-сервера**: Разработать легковесный Node.js сервер
   (Signaling Server), хранящий реестр активных комнат (браузерных хостов).
2. **API списка серверов**: Реализовать отдачу списка серверов с поддержкой:
   - Пагинации (для кнопки «Загрузить ещё»).
   - Фильтрации по регионам (по умолчанию).
   - Отключения регионального фильтра, если общее количество серверов ниже
     порогового значения (например, <= 15).
   - Прямого поиска по имени сервера (игнорируя регионы и пагинацию).
3. **Защита от DDOS пингами**: Внедрить жесткое ограничение (Rate Limiting) на
   Мастер-сервере для количества сигнальных ping-запросов от одного IP.
4. **Маршрутизация WebRTC**: Настроить пересылку SDP-офферов и ответов между
   клиентами и браузерными хостами для установки прямого P2P-соединения
   (STUN обязателен; TURN — опциональный релей по итогам Этапа 0).

### Этап 2: Единое ядро симуляции (Rust -> WASM)

Цель: Перенести всю математику, физику и правила игры в изолированный,
высокопроизводительный «чёрный ящик», который будет работать как на сервере
(хосте), так и на клиенте. Порт ведётся **вертикальными срезами**, а не разом:
сперва один танк проходит весь путь через WASM, затем логика расширяется.

1. **Каркас и вертикальный MVP**: Инициализировать Rust-проект (Rapier 2D
   нативно, без JS-обёртки `@dimforge/rapier2d-compat`), зафиксировать публичный
   API и довести один танк до состояния «ездит через Rust-WASM локально» (шаг
   симуляции + ввод + сборка бинарного кадра + zero-copy чтение из JS).
2. **Перенос серверной логики**: Переписать модули `Game`, `RoundManager`,
   логику оружия (урон, hitscan, таймеры бомб), систему сущностей
   (`ParticipantManager`) и ИИ ботов на Rust.
3. **Перенос клиентской математики**: Перенести механизмы сглаживания сети
   (`SnapshotInterpolator`, `TankPredictor`, `ShotPredictor`) внутрь Rust-ядра.
   Предсказание симулирует **только локальный танк** (как сейчас).
4. **ABI данных и генерация снапшотов**: Определить формат загрузки карт,
   `weapons.js`, `models.js` в WASM; перенести логику `SnapshotPacker` (упаковку
   бинарных кадров) и `unpackFrame` в Rust для прямой работы с памятью.
5. **Тесты ядра**: Rust-тесты для логики + JS↔WASM интеграционный харнесс.
   Учесть, что ~730 текущих тестов частично инвалидируются, а парити-тест
   `TankPredictorParity` теряет смысл (одно ядро вместо двух реализаций).

### Этап 3: Рефакторинг Клиента (Оболочка на JavaScript)

Цель: Превратить текущий клиент (PixiJS) в «глупый» слой ввода-вывода (I/O),
который только отрисовывает данные из WASM и отправляет нажатия клавиш.

1. **Абстракция сетевого слоя**: Заменить использование `ws` (WebSocket) на
   WebRTC Data Channels. Снапшоты (порт 5) идут по unreliable-unordered каналу,
   мета (panel/stat/chat/vote/keyset) — по reliable-ordered. Интегрировать
   «Умный пинг» (клиент пингует только видимые на экране серверы).
2. **Интеграция WASM-модуля**: Настроить передачу пользовательского ввода
   (`InputListener`) напрямую в WASM-модуль.
3. **Связка с рендером**: Изменить конвейер рендеринга (`GameCtrl.parse`).
   Вместо интерполяции внутри JS, PixiJS должен запрашивать у WASM готовый
   плоский массив координат и применять их к спрайтам.
4. **Связка со звуком**: Настроить вызов пространственных звуков по триггерам
   (событиям), которые отдаёт WASM-модуль (например, при взрыве `w2e`).
5. **Обработка разрывов**: Реализовать UI-перехват закрытия WebRTC-канала с
   остановкой рендера и редиректом в Лобби. Явно учесть: **выход хоста = смерть
   комнаты** (host-migration в этой версии нет) — честно показать в UI.

### Этап 4: Разработка Браузерного Хоста (Web Worker)

Цель: Запуск серверной части игры в фоновом потоке браузера создателя комнаты.

1. **Изоляция в Worker**: Разместить загрузку WASM-ядра и игровой цикл (~120 Гц)
   в Web Worker, чтобы не блокировать вкладку. `RTCPeerConnection` при этом живут
   в главном потоке (в Worker нельзя), который выполняет роль роутера пакетов —
   это же позволяет убивать Worker без разрыва P2P (основа Worker Handoff).
2. **Инициализация настроек**: Настроить передачу параметров (имя, карта, лимит
   игроков, таймеры), введённых пользователем на сайте, в Worker при его запуске.
3. **Разделение ролей (опционально, UX)**: Вкладка-создатель может служить
   только панелью управления. При анти-чите «минимум» это чисто удобство, не
   мера защиты.

### Этап 5: Бесшовные обновления и соц-модерация

Цель: Доставка кода и данных без потери игроков + минимальная модерация.

1. **Динамическая загрузка карт (Обновление данных)**: Удалить жёсткую привязку
   карт к сборке. Настроить Worker на скачивание актуальных JSON-карт с
   Мастер-сервера по сигналу `update_available` (для применения в следующем
   раунде).
2. **Обновление кода**: Реализовать по результату Этапа 0. Если Worker Handoff
   подтверждён — эстафетная передача состояния между Worker'ами без разрыва
   WebRTC. Если нет — fallback: обновление на границе карты с коротким reconnect.
3. **Социальная модерация (`/ban`)**: Единственная анти-чит-мера. Команда в чате
   с обязательной причиной; Мастер-сервер исключает хоста из выдачи при
   достижении лимита уникальных жалоб.
4. **Гигиена среды**: Строгий CORS/Origin на Мастер-сервере, CSP на `.wasm`,
   минификация/обфускация JS-оболочки. Это отсекает «уличных» злоумышленников, но
   **не** хоста-читера — см. [Модель угроз](P2P-PLAN-SAFITY.md).
