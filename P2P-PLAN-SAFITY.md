Защита от подмены исполняемого файла (WASM) на стороне хоста.

Поскольку сервер физически находится на устройстве пользователя, нельзя просто
запросить у пользователя хэш (контрольную сумму) файла: модифицированный клиент
легко отправит "правильный" хэш оригинального файла, продолжая исполнять
читерский код.

Игра строго детерминирована (одинаковый ввод при одинаковых условиях всегда дает
одинаковый результат — это свойство физики Rapier и единого ядра Rust), можно
использовать архитектурные паттерны для выявления подмены.

Вот варианты, которые можно рассмотреть:

### 1. Теневой консенсус (Shadow Consensus / Cross-Validation)

Мастер-сервер перекладывает проверку на самих игроков.

- **Идея:** Поскольку все обычные клиенты (игроки) скачивают тот же самый
  оригинальный WASM с вашего сайта для предсказания движений, их можно
  использовать как «аудиторов».
- **Как реализовать:**
  1. Мастер-сервер тайно назначает 1-2 случайных игроков в комнате
     «Валидаторами». Хост не знает, кто именно является валидатором.
  2. WASM-клиент валидатора в фоновом режиме прокручивает не только свой танк,
     но и полноценную симуляцию сервера на основе тех координат и вводов,
     которые он получает.
  3. Каждые 10 секунд Хост и Валидаторы вычисляют легковесный хэш состояния игры
     (например, сумму координат всех танков и их здоровья) и отправляют его на
     Мастер-сервер.
  4. Если хэш Хоста не совпадает с хэшем Валидатора (например, Хост сделал себя
     бессмертным, а Валидатор посчитал, что Хост должен был получить урон) —
     Мастер-сервер понимает, что хост играет по другим правилам (использует
     подменный WASM).
  5. Комната немедленно закрывается, а IP хоста банится на уровне
     Мастер-сервера.

### 2. Выборочные проверки реплеев (Spot-checks)

Проверку может делать Мастер-сервер, но не в реальном времени, а выборочно.

- **Идея:** Мастер-сервер (Node.js) сам загружает оригинальный `vimp_core.wasm`
  (работающий без графики, headless).
- **Как реализовать:**
  1. Хост обязан раз в несколько минут (или при подозрительных событиях,
     например, множественных убийствах) отправлять на Мастер-сервер компактный
     «дамп ввода» — список всех нажатий клавиш всеми игроками за последние 5
     секунд + начальное состояние физики.
  2. Мастер-сервер скармливает эти вводные данные своему эталонному WASM-ядру.
     Поскольку симуляция идет без задержек сети (в цикле `while`), 5 секунд игры
     просчитываются на сервере за миллисекунды.
  3. Мастер-сервер сравнивает итоговое состояние своего эталонного расчета с
     тем, что заявил Хост. При несовпадении — бан.

### 3. Криптографическая ротация ключей (Dynamic Salts)

Метод ломает жизнь читерам, которые пытаются написать кастомный клиент,
заставляя их обновлять свой чит каждый день.

- **Идея:** Изменить протокол общения так, чтобы поддельный клиент просто не мог
  собрать валидный сетевой пакет без знания временного секрета.
- **Как реализовать:**
  1. При создании сервера Мастер-сервер выдает хосту уникальный
     криптографический «соль» (токен), который меняется каждые 15 минут.
  2. Внутри оригинального WASM-модуля (на стороне хоста) вшита логика: перед
     отправкой бинарного снапшота клиентам (порт 5), WASM подмешивает эту соль в
     генерацию структуры пакета или добавляет в конец пакета хэш-подпись (HMAC).
  3. Клиенты, получая пакет, проверяют эту подпись через свой WASM (они тоже
     получают текущую соль от Мастер-сервера).
  4. Читеру придется не просто переписать здоровье или урон, ему придется
     вытаскивать из вашего WASM криптографический алгоритм подписи и
     реализовывать его в своем кастомном сервере. Это резко повышает порог входа
     для создания читов.

### 4. Защита среды исполнения (Environment Lock)

Базовые меры веб-безопасности, которые не пустят 90% злоумышленников "с улицы".

- **Идея:** Запретить модифицированным браузерам или локальным скриптам
  связываться с вашей инфраструктурой.
- **Как реализовать:**
  1. **Strict CORS & Origin:** Мастер-сервер отклоняет любые
     WebSocket-соединения, если заголовок `Origin` не совпадает с вашим
     официальным доменом (например, запретить подключения с `localhost` или
     `file://`).
  2. **Content Security Policy (CSP):** Настроить HTTP-заголовки так, чтобы
     браузер отказывался загружать любой `.wasm` файл, который пришел не с
     вашего официального CDN.
  3. **Обфускация:** Скомпилированный Rust WASM по умолчанию сложно читать, но
     читеры могут написать JS-обертку (перехватчик между WebRTC и WASM). Поэтому
     весь JS-код (оболочку) нужно жестко минифицировать и обфусцировать
     (превратить переменные в нечитаемый мусор).

### 5. Жалоба пользователей

- **Через команду `/ban` в чате:**
  1. В интерфейсе клиента (`Chat.js`) добавить команду `/ban`, после команды
     нужно обязательно указать причину бана. Команда не отображается публично.

  2. Команда отправляется на Мастер-сервер с ID хоста и ID игрока (чтобы один
     игрок не накрутил жалобы).

  3. Логика Мастер-сервера: если у хоста накапливается `X` уникальных жалоб за
     последний час (например, 5 жалоб при лимите в 10 игроков), поле `status`
     меняется на `banned`.

  4. Мастер-сервер прекращает отдавать этот сервер в ответе `GET /servers` и
     разрывает сигнальный WebSocket. Читер остается в своей комнате один, новые
     игроки к нему не попадут.
