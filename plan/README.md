# План: разделение на два репозитория + авторизация на лобби

Два независимых, но пересекающихся направления. Направление A (`split_*.md`)
— вынос игры в отдельный репозиторий и публикация движка как npm-пакета +
Rust-crate. Направление B (`auth_*.md`) — OAuth-авторизация, глобальный ник,
rank и состояние игрока через центральный auth-сервис.

Выполненные планы лежат в [`plan/done/`](done/), каждый в своей папке.
Архив предыдущего плана (разделение движка/игры на рантайм-уровне, Этапы 1–8 +
доработки Д1–Д7 по итогам кодревью) — [done/engine-game-split/](done/engine-game-split/).

**Зафиксированные решения (ответы пользователя):**
- Подключение игр — **готовые npm-бандлы** (игра сама собирает `dist/`,
  включая WASM, в своём CI и публикует пакет).
- Аккаунты/рейтинг — **центральный auth-сервис** с общей БД PostgreSQL, все
  мастера ходят к нему по API.
- Ник/rank — **JWT + отчёт хоста**: мастер выдаёт подписанный токен, хост
  проверяет подпись и берёт ник из токена, итоги матча хост шлёт на мастер
  аутентифицированным REST.

## Контекст

Проект уже прошёл внутреннее разделение движка и игры: граница
«движок↔игра» — это **URL-driven плагин-контракт**. Движок нигде не импортирует
игру статически; он грузит её только динамически по манифесту:
`import(manifest.entries.client)` в клиенте, `import(room.game.hostEntryUrl)` в
Worker хоста, `GameCatalog` на мастере читает `dist/manifest.json`. Барьер
ESLint (`no-restricted-imports`) и Rust-трейты `GameDef`/`GameSim`/
`GameClientDef` уже разводят код. То есть **рантайм-граница готова** — не хватает
только «физического» разъезда по репозиториям и упаковки движка в публикуемые
артефакты.

Авторизации сегодня нет вообще: у мастера нет ни БД, ни сессий, ни OAuth. «Ник»
самозаявляемый — вводится в игровую форму (`auth.pug`, схема из
`games/tanks/src/config/auth.js`), проверяется хостом (`validateAuth` →
`ParticipantManager.createHuman.checkName`, дедуп только внутри одной комнаты) и
хранится лишь в `localStorage['userName']`. Мастер ник игрока не видит вообще.
Статистика (`Stat.js` / `RoundManager.reportKill`) эфемерна, обнуляется каждый
раунд, нигде не сохраняется. `/rank`, ELO, «скиллов» — нет.

## Направление A. Разделение на два репозитория

**Цель.** Игра (`games/tanks/`) выезжает в отдельный репозиторий и подключается
к движку как опубликованный npm-пакет с готовым `dist/`. Движок становится
самодостаточным приложением, публикующим два артефакта: npm-пакет
`@vimp/engine` и Rust-crate `vimp-engine-core`.

```
Репозиторий движка (engine)                 Репозиторий игры (напр. vimp-tanks)
──────────────────────────                  ───────────────────────────────────
publish: @vimp/engine (npm)   ◀── depends ── package.json: "@vimp/engine": "^x"
publish: vimp-engine-core (crate) ◀─ dep ── core/Cargo.toml: vimp-engine-core = "x"
                                            CI: core:build:web + game:build → dist/
config/master.js: games: [                  publish: @vimp/tanks (npm, ships dist/)
  { id:'tanks', package:'@vimp/tanks',
    version:'1.2.3' }, ...]
master деплой: npm i игровых пакетов → node_modules/@vimp/tanks/dist/
GameCatalog читает манифесты установленных пакетов (вместо скана games/*)
```

Рантайм-контракт (манифест, `/games/*` эндпоинты, `express.static`, динамический
`import()`, композитный `codeVersion`) **не меняется** — меняется только то,
*откуда* берётся `dist/`.

### Верификация (A)

1. Локально: `npm link @vimp/tanks` (или локальный path) → `npm run dev` →
   лобби видит игру, создаётся комната, две вкладки играют (движение/выстрел/
   панель/звук) — как сегодня.
2. `npx eslint .` (движок и игра по отдельности), `npm test` в обоих репо,
   `npm run core:test` в репо игры — зелёные.
3. Сборка движкового Docker-образа **без** Rust-тулчейна; игра ставится из
   registry; runner отдаёт `/games/tanks/manifest.json` и ассеты.
4. Проверка контракта версий: собрать игру со `engineApi`, не равным
   `ENGINE_API_VERSION` мастера → игра отклоняется с внятной ошибкой.

## Направление B. Авторизация на лобби-сервере

**Цель.** Вход через Google/Apple/GitHub (без пароля). При первом входе игрок
придумывает глобально-уникальный ник (один на все игры). Ник — источник для
рейтинга (`/rank`) и для сохранения состояния игрока (скиллов). В игре ник не
создаётся: игрок приходит с ником из лобби. Rank и состояние авто-подгружаются на
любом хосте и возвращаются на лобби по ходу игры.

```
Central Auth Service (новый, свой домен, PostgreSQL)
  • OAuth google/apple/github (владеет client_id/secret/redirect)
  • users:      id, provider, provider_uid, nick(UNIQUE), created_at
  • ratings:    user_id, game_id, rank, updated_at
  • states:     user_id, game_id, state(JSONB opaque), updated_at   ← «скиллы»
  • выдаёт короткоживущий JWT { sub:user_id, nick, iat, exp }, подписанный своим ключом
  • REST: /oauth/*, /nick (первый вход), /rank?game=, /state?game=, /jwks

Master (лобби, каждый домен)         Host (браузер создателя, недоверенный)
  • редиректит на Auth Service        • верифицирует JWT по /jwks мастера
  • отдаёт JWT клиенту                • берёт ник из токена (не свободный ввод)
  • проксирует /jwks, /rank, /state   • на join запрашивает state+rank игрока
  • CSP разрешает провайдеров         • в конце матча/раунда POST итогов на мастер

Client
  • логинится → хранит JWT
  • несёт JWT в webrtc_offer / AUTH_RESPONSE
  • /rank — читает рейтинг с мастера
```

Оговорка (важно зафиксировать в доках): хост — недоверенный браузер, поэтому
любые присылаемые им rank/скиллы **технически подделываемы**. Это ограничение
P2P-архитектуры; полноценный анти-чит вне рамок данного плана. JWT защищает
идентичность (ник нельзя подменить), но не защищает от накрутки результата
своего матча.

### Верификация (B)

1. Локальный smoke: логин через одного провайдера (dev-конфиг), выбор
   уникального ника, повторная попытка занятого ника → отказ.
2. Вход в игру: ник в игре совпадает с ником лобби; свободный ввод имени
   недоступен; поддельный/просроченный JWT → хост отклоняет.
3. Сыграть матч → rank/state обновились на auth-сервисе; переигровка на **другом**
   домене-мастере → подгрузились те же rank/state (проверка «глобальности»).
4. `/rank` в чате печатает актуальный рейтинг.
5. `npx eslint .`, `npm test`, `npm run core:test` — зелёные; интеграционный
   тест auth-сервиса (nick uniqueness, JWT verify, rank upsert).
6. CSP: проверить, что OAuth-редиректы проходят, а политика не ослаблена сверх
   необходимого.

## Порядок и зависимости между направлениями

- A и B в значительной степени независимы. B2–B4 трогают движковый хост/клиент —
  их удобно делать **до** физического разъезда (A3), пока код в одном дереве.
- Игро-специфичные части B (схема state/rank в `game.js`, `/rank`, правка
  `auth.js`) после A3 живут уже в репозитории игры.
- Рекомендуемая последовательность: **B2–B5** (auth в монорепо) → **A1–A2**
  (артефакты движка + конфиг игр) → **A3–A5** (разъезд + доки) → добить **B6**
  и игровые части B в новом репо игры.
- Работа начинается с B1–B2 (см. решение пользователя от 2026-07-24): сначала
  авторизация, физический перенос игры (A3) не начинать раньше времени.
